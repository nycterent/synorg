## Code Review: `fix/crash-safe-reclaim` (audit P0-3)

**Overall:** The core mechanism is sound. Mark-before-cordon is the right ordering, the resumer's selection triple (cordoned + lent + marker) is correct, and the primary crash case — process death *after* cordon on EKS with a live NodeClaim — is recovered cleanly. The jq is correct, and the RBAC/test-parser wiring is right. Two real correctness gaps and one destructive test-safety gap keep it from being a clean landing. Details below, ranked.

### 🔴 1. NodeClaim delete failure → resumer hot-loops forever (re-drain every tick)

**`reconcile.sh:286-291`** (root cause) + **`reconcile.sh:327-335`** (amplifier)

`reclaim_node` is always invoked with errexit suspended — `if reclaim_node ...` (resumer:327, reconcile_taints:379) and `reclaim_node ... || true` (waves:586). Bash disables `-e` throughout a function body called in those contexts. So when `kc delete nodeclaim "$ncl" --wait=false` (line 287) fails, execution does **not** abort — it falls straight through to:

```
log info action=nodeclaim_deleted ... reason=NodeScrubStarted   # line 288 — falsely logs success
emit_event NodeScrubStarted ...                                  # line 289
return 0                                                          # line 290
```

reclaim_node returns **0** on a *failed* delete. The resumer (327-328) then treats it as success, does **not** clear the marker, and the node — which still exists because the delete failed — is re-selected next tick. Result: an unbounded every-tick re-drive (each doing a fresh drain up to `grace*3` = 360s, a fresh failing delete, and fresh ReclaimWaveStarted/NodeDraining/NodeScrubStarted events).

**Failure scenario:** morning-ramp API pressure (exactly when waves fire) causes the `--request-timeout=30s` delete to time out. The resumer re-drains the same node every tick indefinitely. This is the specific case you asked about ("a live NodeClaim that repeatedly fails to delete") — and it is a perpetual loop, not a one-shot.

The fall-through is pre-existing in `reclaim_node`, but the diff newly weaponizes it: waves fire once/day and close fires once/close, whereas the resumer runs **every tick**. Fix: guard the delete (`if kc delete ...; then ... return 0; fi`) so a failed delete returns 1 (degraded) rather than a false success — then the resumer's degraded branch bounds it. (This also fixes the false `nodeclaim_deleted` log across the whole controller.)

### 🟡 2. Degraded resumer branch re-creates the exact P0-3 strand — silently

**`reconcile.sh:329-335`**

On a degraded return (rc 1), the resumer clears the marker and leaves the node **cordoned + lent + unmarked**. That state is skipped by *every* path:
- resumer (319) requires the marker — gone;
- `reconcile_waves` (566) `select(.spec.unschedulable != true)` — excludes cordoned;
- `reconcile_taints` close path (376) `[ "$cordoned" != "true" ] || continue` — skips cordoned.

So the node sits lent forever pending manual operator action — which is precisely the P0-3 condition the fix targets, now re-created with the marker removed (so it's not even re-discoverable by the resumer). The clear is silent (`&gt;/dev/null 2&gt;&amp;1 || true`) — no event, no operator-facing log beyond the initial `reclaim_resumed`.

This also diverges from the established degraded semantics: `reconcile_taints`' degraded close path (383-386) **untaints** the node (NodeReturnedToProd) so capacity is returned. The resumer does not. The code comment at 330-333 ("the same fate the degraded reclaim path already assigns it") is therefore inaccurate — the close path untaints; this path leaves it lent.

Trigger frequency: **always on kind** (log-only reclaim always returns 1) and on EKS whenever no NodeClaim is found. Recommend either mirroring the close-degraded path (untaint to return the node) or keeping the marker + emitting a loud NodeReturnedToProd/degraded event so the abandoned node is not indistinguishable from the original bug. Whichever you choose, it should be a deliberate, documented decision and must not be silent.

### 🟡 3. Test scenario 9 has no Karpenter-skip guard — destructive on an EKS context

**`test.sh:421-435`**

Scenarios 6, 7, and 8 each guard with `if k api-versions | grep -q '^karpenter.sh/'; then skip ... else`. Scenario 9 omits it. The live tier accepts any context via `LENDING_TEST_CONTEXT`. On a Karpenter cluster, scenario 9 does `k cordon` + `k annotate` on a real node, then ticks — the resumer runs `reclaim_node` with `karpenter=1`, which performs a **real cordon + drain + `kc delete nodeclaim`**, terminating the live test node (a GPU node). The `reclaim_resumed` assertion still passes, masking the destruction.

Add the same Karpenter-present skip the sibling scenarios use, or restructure scenario 9 to assert selection without driving a real termination.

### 💭 4. Crash between annotate and cordon is not resumer-recoverable (comment overstates)

**`reconcile.sh:276-281`**

A crash in the one-statement gap between `kc annotate` (280) and `kc cordon` (281) leaves the node **marked + lent + uncordoned**. The resumer requires cordoned (317), so it will not pick it up. The comment (276-278) says "the normal wave/taint path still selects it" — true for selection, but the wave path is gated by the once-per-day fired-marker (waves:560-561). If that wave already fired, only `reconcile_taints` at the next window **close** recovers it. So the node can stay lent until close rather than being re-driven promptly.

Not a permanent strand (close recovers it, and the crash window is two adjacent kubectl calls), so low severity — but the comment's recoverability claim is stronger than reality. Worth a one-line correction.

---

### 💭 5. Churn on successful async delete (no "already terminating" guard)

**`reconcile.sh:324-328`**

`--wait=false` means a successfully reclaimed node lingers (finalizers / instance teardown) as cordoned+lent+marked across several ticks. The resumer re-selects and re-drives it each tick — re-drain, re-issue delete (idempotent), and re-emit ReclaimWaveStarted/NodeDraining/NodeScrubStarted — until the Node object disappears. Bounded by Karpenter finishing, but noisy (event spam + redundant drains). A cheap guard (skip nodes whose NodeClaim already carries a `deletionTimestamp`, or whose node has one) would eliminate it.

### 💭 6. Scenario 9 assertion is weaker than it looks

**`test.sh:431`**

It only greps `action=reclaim_resumed`, which is logged (326) **unconditionally before** `reclaim_node` is called — so it proves selection but not that the node entered the reclaim path. Strengthen by also asserting `reason=resume_incomplete` and (kind) `reclaim_intent`, matching the rigor of scenarios 6/7. There is also no negative control demonstrating the node *would* be skipped without the resumer.
